# Contributing

Thanks for considering a contribution. Pull requests are welcome —
genuinely so, this fork only exists because the upstream forks went
quiet and I'm not running it full-time either.

A few things worth knowing up front.

## About the maintainer

I maintain this fork in my spare time around a full-time day job, so
I can't always reply to issues or review PRs within a few days. If a
PR sits for a while, it isn't rejection — it usually just means the
week got away from me. Feel free to bump it with a polite ping.

For quick questions, you'll reach me faster via:

- Pale Moon forum — user **sinfulosd**
- GitHub Issues on this repository
- Private DMs for anything security-related

## What gets merged quickly

Contributions in these categories have the easiest time getting in:

- **Bug fixes** with a clear "before / after" description, ideally with
  a minimal reproduction script or a link to the issue.
- **Translations** — either filling in untranslated strings in an
  existing `locale/<code>/` directory, or adding a new locale. English
  placeholders in non-en-US files are meant to be replaced; that's a
  great first PR.
- **Small, self-contained features** that don't alter core sandbox /
  script-loading / IPC behaviour. Things like an extra
  Add-ons Manager button, a new sort order, a GM API polyfill, an
  opt-in preference — those are low risk.
- **Documentation** improvements to `README.md`, `CHANGELOG.md`, or
  this file.

For all of the above, the only hard requirements are:

1. Use the browsers this fork targets (Pale Moon, Basilisk, or a
   reasonable equivalent) to verify your change actually loads and
   does what you think it does before opening the PR. UXP is not
   modern Firefox; the only way to catch regressions is to run the
   build.
2. Keep the scope of a PR focused. One bug fix or one feature per PR
   is much easier to review and revert than five unrelated changes.
3. Match the existing code style — tabs/indentation, brace style,
   JSDoc comments on non-trivial functions — so the diff stays small.

## What needs a discussion first

Please open an Issue *before* starting work if your PR would touch any
of these:

- **Core sandbox or script-loading architecture** — anything in
  `modules/sandbox.js`, `modules/menuCommand.js`, `modules/ipcScript.js`,
  or the injection pipeline. These files have subtle cross-compartment
  and XrayWrapper invariants that aren't obvious from reading a single
  function. A wrong change here silently breaks userscripts on some
  browsers and not others (as issue #13 recently demonstrated).
- **Changes to the extension ID** or the `update.rdf` format.
- **New GM API surface** — adding `GM_*` / `GM.*` functions that
  scripts can grant. Design the API first, implement second.
- **Refactors that touch more than ~5 files** in a way that isn't a
  pure find-and-replace.
- **Anything that could regress Pale Moon 28 compatibility** — old
  SpiderMonkey builds have quirks (decompiler bugs, strict Xray policy,
  no template literals in some versions) that modern code style often
  trips over.

An Issue describing the plan lets me spot red flags before you invest
hours in code I might have to ask you to rework.

## How to open a PR

The usual workflow is fine:

1. Fork the repository on GitHub.
2. Create a feature branch from `master` (please not from an old
   release tag).
3. Commit with a descriptive message. The first line should summarise
   the change in ≤ 72 characters; a short paragraph below explaining
   "why" is appreciated but not mandatory.
4. Open the PR against `SecondCityOsD:master`. Include:
   - A one-line description in the title.
   - A short body explaining the problem and the fix.
   - The browser(s) you tested on and the version number.
5. Respond to review comments at your own pace — same latency caveat
   applies both ways.

## What I can't accept

A few things will bounce regardless of how well the code is written:

- **Obfuscated or minified code** that isn't generated by a tool in
  the repository's build pipeline.
- **New binary blobs** (images, precompiled libraries, fonts, …)
  without a clear source link and a license that's compatible with
  MPL 2.0.
- **Telemetry, analytics, or any kind of phone-home** code. This
  extension doesn't report anything; it shouldn't start.
- **Breaking changes** to behaviour users already depend on, unless
  there's a documented migration path (see the 3.5.0 → 3.6.0 ID
  change for an example of how I prefer to handle those).

## If you're just reporting a bug

Open an Issue with:

- Browser name and version (e.g. "Pale Moon 33.3.1" or "New Moon
  28.10.7a1 (32-bit) (2026-04-10)").
- Operating system.
- Greasemonkey for UXP version.
- A minimal userscript that reproduces the problem, if the bug is
  script-triggered.
- Anything relevant from the Browser Console (Ctrl+Shift+J).

A clear report with a reproduction is almost always faster to fix
than a patch against a bug I can't reproduce.
